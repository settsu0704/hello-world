# ソフトウェア工学最終課題

### 第1.2回授業
ソフトウェアの定義とは
1. 実行されることによって必要な特性、機能、性能を提供する命令語群
2. プログラムが適切に情報を扱うことを可能とするデータ構造
3. プログラムの操作や使用法を記述した情報

の3つからなる。\
　ソフトウェアは新しい環境や技術のニーズを満たすように適用しなければならいという特徴をもつ。\
　ソフトウェア開発全体で意識すべき基本原則として「Hookerの7原則」がある。\
　近年、ソフトウェア利用の変化・多様化によりソフトウェア需要は増加している一方でIT人材の不足や世の中の大規模化や複雑さによりソフトウェア工学の必要性は高まっている。

### 第3回授業
ソフトウェアライフサイクルはソフトウェアの企画から開発・運用、廃止までの一連の流れのことである。\
**要件定義**\
　要件定義とは実現する機能・実現しない機能を明確にすることである。要件定義を行うことで、スコープの共有や社内外からの協力を依頼しやすくなる。\
**設計**\
　要件定義書を設計書に落とし込む。WBSなどが用いられることもある。\
**制作**\
　設計書に従ってシステム構築を行う。自社で開発を行う「内製」と開発を外部委託する「調達」がある。\
**テストとデバック**\
　バグのないソフトウェアを作ることは不可能であるためあらかじめテストしバグを早期に発見・対処することが理想である。

### 第4回授業
　プロジェクトとは有期性と独自性をもった仕事である。\
　プロジェクトを達成するためには目標に対して何が必要か計画し達成に向かう「バックキャスティング」が用いられる。\
　ソフトウェアには様々な開発手法がある。\
**ウォーターフォール型**\
　要件定義から順に行う方法で進捗管理が容易である一方で後工程にしわ寄せが集中するリスクもある。\
**スパイラルモデル**\
　プログラム開発を小さなフェーズに分割する方法で、想定外の作業量が発生するリスクがある。\
**アジャイル開発**\
　開発の流れを何度も繰り返し行う開発方法である。変化に対応して無駄を廃し、最適な手法で動くソフトウェアの提供を優先している。\
　これらの他にも「スクラム」や「カンバン」などの開発手法がある。

### 第5,6回授業
　WBSとはプロジェクト目標を達成し、必要な要素成果物を生成するためにプロジェクトチームが実行する作業を、要素成果物を主体に階層的に要素分解したものである。\
　WBSでは作業を細かく分解し、作業順に並べる必要がある。\
　WBSには全体管理と作業計画が明確化されることやプロジェクト実施時にはWBSに則り実行するのみとなるなどのメリットがある。\
　WBSを作る際はまずスコープを明確にする。その後、データ収集やデータ分析など作業をグルーピングする。グルーピングした作業の相互関連を考え、最後に各グループの作業を洗い出す。

### 第7回授業
　コーディングは書くよりも読まれることの方が多い。そのためソフトウェア開発において読みやすいコードを書くことが重要である。\
　また他人が読みやすいだけでなく、将来の自分が読みやすいようにルールを設定することが必要である。\
　以下にPEP8でのルールの例を示す。\
**1行の長さ**\
79文字以内にする。\
**スペース**\
演算子の前後にスーペースを前後に1つずつ開ける。\
**改行**\
演算子の位置をそろえる。\
　またチェックツールとして「flake8」などがある。

### 第8,9,10回授業
 　作ったアプリケーションではバージョン管理が必要となる。\
 　ファイルを「誰が」、「いつ」、「どのように」変更したのかを管理する。また管理の方法には1つのリモートリポジトリで管理する「集中管理型」とローカルレポジトリで管理する「分散管理型」がある。\
 　分散管理型のバージョン管理システムに「git」という代表的なものがある。\
 　gitでは変更履歴を残したり、変更した箇所に戻ったりすることができる。また他人と共同編集することもできる。\
 　gitにはプロジェクトを管理するレポジトリと呼ばれるフォルダがあり個々の「ローカルレポジトリ」と共有で管理する「リモートリポジトリ」がある。\
 　一般的にはローカルレポジトリでcommmitした変更をリモートリポジトリに反映させる(git push)とリモートリポジトリにある内容をローカルレポジトリに反映させる(git pull)の機能がある。\
 　またブランチでは作業を枝分かれさせることができ、共同作業・平行作業が可能となる。\
 　それぞれが自分のブランチで行った作業をmainブランチにマージして開発を進めることを「gitフロー」と呼ぶ。

 ### 第11回授業
githubには様々な機能がある。

**fork**\
他人のリポジトリを自分用にコピーし、そのプロジェクトを引き継いで開発・改善するための仕組み\
**issues**\
レポジトリに関する問題・課題・バグ・機能追加や質問などを挙げるための機能\
**progects**\
issuesなどを管理するためのテーブル・kanba

　簡単に文章を記述するための記法として「マークダウン」がある。\
　githubなど多くのアプリケーションで採用されており.md拡張子を用いる。\
　pandocを使うとtexや.pdf、.hemlんどに変換することができる。

### 第12,13回授業
githubにはコード変更を恐竜リポジトリに頻繁に統合するCIというプロセスがある。CIの自動化により開発のスムーズな進行を促進する。\
　またコード変更をテスト環境や本番環境に自動的にデプロイするCDというプロセスもある。これによりユーザーフィードバックを迅速に反映できる。\
　CI/CDパイプラインの基本的なステップは以下のようになっている。\
1. ソース：コード変更をトリガーにワークフローを起動
2. ビルド：ソースコードをコンパイル
3. テスト：自動テストを実行
4. デプロイ：テスト済みのコードを本番環境にデプロイ
5. 検証：デプロイされたアプリケーションの動作確認
6. モニタリング：本番環境での継続的な監視
